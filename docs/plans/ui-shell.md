# ledgerboy — UI shell

## Status: LOCKED — copy from tonearmboy, do not redesign

The user's directive, verbatim: *"look at tonearmboy the colorful material 3e and also the open source licenses etc should all be in the same style without me having to reprompt you 50x."*

The UI shell is **not a research-round decision**. It is locked. Subagents working on ledgerboy must not redesign these surfaces. They copy the patterns from tonearmboy and adapt the *content* (destinations, settings entries, About strings), not the shape.

The shell is what makes the family feel like a family. Net-worth charts on top of a tonearmboy-shaped settings catalog tells the user "this is part of the same set."

The pattern: **catalog DSL → section entries → render**. Adding a new settings entry means writing one entry in a `…Entries.kt` file; the DSL handles the rest.

### Theming

See [`m3-expressive.md`](m3-expressive.md) for the locked M3E theming approach. The shell files above are written against `MaterialTheme.colorScheme.*` and `MaterialTheme.typography.*` tokens — once the theme file is in place, the shell colours and types follow automatically.

## What NOT to do

- **Do not invent new navigation patterns.** No bottom-sheet navigation, no drawer, no horizontal tabs at the top, no custom rail. Vertical navigation rail + top bar — that is the shape across the family.
- **Do not invent new settings shapes.** No "settings as a separate fullscreen activity," no "settings as a dialog," no "settings as a horizontal pager." The settings catalog DSL is the pattern.
- **Do not reskin AboutScreen / LicensesScreen.** Their shape is the family's shape. Update strings only.
- **Do not pull in a third-party charts-on-shell library** that wants to own the navigation. Charts live inside destinations, not as the shell itself.

## Phases that depend on this

- `main.md` **Phase A.5 / A.6 / A.7** — the actual file copy + rename happens during scaffold.
- Every subsequent phase that adds a destination (`Accounts`, `Budgets`, `Assets`, `Charts`) lands a new destination inside the existing rail; it does not modify the rail.
- Every phase that adds a setting (FX source, biometric gate, banking-credential management, base currency) lands a new entry in one of the `…Entries.kt` files; it does not modify the catalog DSL.
